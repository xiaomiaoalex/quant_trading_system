# Rules 分层架构

本目录采用 **TRAE 四层规则架构**，实现上下文成本的极致优化和无限的扩展性。

## 层级说明

| 层级 | 文件 | 作用 | 生效方式 |
|------|------|------|---------|
| **L0** | `L0_COLLABORATION.md` | 协作/输出规范 | 始终生效 |
| **L1** | `L1_TECH_STACK.md` | 技术栈/工程规范 | 按文件生效 |
| **L2** | `L2_BUSINESS.md` | 业务/领域规则 | 智能生效 |
| **L3** | `L3_WORKFLOW.md` | 工作流/SOP | 手动触发 |

## 核心原则

### 分层设计的好处

1. **修改不影响其他层级**：改L1不影响L0/L2/L3
2. **全局Rules跨项目共享**：L0/L1可复用
3. **模块级Rules满足特殊需求**：L2/L3可定制

### 规则治理三原则

1. **新增前想清楚分层**：这条Rule应该放在哪一层？影响范围是什么？
2. **持续带偏则降级/删除**：如果AI经常因为某条Rule而出错，说明Rule本身有问题
3. **时机不对改生效方式**：不是Rule不好，可能是触发时机不对。L2改L3，或者反过来

## 渐进式披露架构

```
┌─────────────────────────────────────────────────────────────┐
│  元数据层（Metadata）- 始终加载                              │
│  L0_COLLABORATION.md  (~50 tokens)                          │
│  AI拥有"能力清单"的概览                                     │
├─────────────────────────────────────────────────────────────┤
│  核心指令层（Instructions）- 按需加载                         │
│  L1_TECH_STACK.md / L2_BUSINESS.md / L3_WORKFLOW.md        │
│  根据任务上下文智能加载                                     │
├─────────────────────────────────────────────────────────────┤
│  外部资源层（Resources）- 按需动态读取                       │
│  skills/ 目录下的具体文件和配置                             │
│  复杂的参考信息仅在必要时查阅                               │
└─────────────────────────────────────────────────────────────┘
```

## 与主规则入口的关系

- `.traerules` / `AGENTS.md` / `CLAUDE.md` 是工具入口
- `rules/` 目录是规则的具体实现
- 工具入口引用 `rules/` 目录中的规则

## 维护指南

### 添加新规则

1. 确定规则属于哪一层：
   - L0：协作方式、输出格式、沟通语言
   - L1：技术选型、代码风格、命名规范
   - L2：业务约束、领域模型、架构边界
   - L3：流程步骤、SOP、检查清单

2. 写入对应文件

3. 更新本 `README.md` 的版本记录

### 修改现有规则

1. 检查修改影响范围
2. 同步更新 `.traerules` / `AGENTS.md` / `CLAUDE.md`（如果需要）
3. 更新版本记录

### 删除规则

1. 确认规则是否被引用
2. 从对应层级文件中删除
3. 更新版本记录
