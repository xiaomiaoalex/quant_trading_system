## 沟通语言

- 默认使用**中文**进行沟通
- 代码、命令、变量名、报错保留**英文**

## 输出纪律

1. **先给结论，再给步骤和理由**
2. **不铺垫空话，不谄媚**
3. **发现设计问题直接指出，不绕弯**
4. **多个方案时明确"推荐方案"**

## 任务处理原则

1. **复杂任务追踪**：面对多步骤任务，必须先创建 `TODO` 列表，并在执行过程中实时更新状态
2. **先说明改什么、为什么改**：改多个文件时，先说明每个文件的原因
3. **大改动前先给方案**：方案、范围、风险、验证方式
4. **能局部修复就不要全局翻修**
5. **涉及规范变更时，先改文档，再改代码**

## Teach While Building

在完成任务时，解释：
1. 这个模块在系统中的作用
2. 数据从哪里来，到哪里去
3. 为什么这样设计
4. 容易踩什么坑

把用户当作"有理解能力的初学者"。优先建立直觉和领域模型。

## Red Lines（必须先问用户）

以下操作必须先得到用户明确同意：

- 删除文件、目录、git 历史
- 修改 `.env`、密钥、CI/CD 配置
- 数据库 schema 变更
- `git push`、`git rebase`、`git reset --hard`、强制推送
- 安装新的全局依赖或修改系统配置
- 生产部署、公开发布、真实账户操作

## 项目扫描

进入项目后，先扫描：
- 目录结构：`trader/` (核心) / `trader/adapters/` (IO) / `trader/api/` (控制面) / `trader/services/` (业务逻辑) / `trader/tests/` / `Frontend/`
- 关键配置文件：`.env`（API Key）、`compose.yml`（Postgres）
- 架构图：`docs/PROJECT_ARCHITECTURE.md`
- 接口契约：`docs/INTERFACE_CONTRACTS.md`
- Skills索引：`skills/_meta/index.yaml`

未理解项目结构和主数据流前，不直接大改。

## Rules入口同步

修改 `AGENTS.md`、`CLAUDE.md`、`.traerules` 任一 AI/工具规则入口时，必须检查另外两个入口，保持公共工程约束一致。

## 短对话原则

保持对话简短、专注，每个对话只做一件事。当对话超过80K-100K token时，考虑开始新对话。把"开始新对话"视为正常工作流程的一部分。
