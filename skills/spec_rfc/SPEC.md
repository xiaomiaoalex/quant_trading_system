# Spec RFC Skill
# 版本: v1.0.0

## Metadata

- **Skill Name**: spec_rfc
- **触发场景**: 复杂需求实现、跨模块重构、新增外部依赖、多服务交互
- **Token成本**: ~300 tokens（核心指令）
- **触发条件**: 任务涉及以下任一场景时必须执行此Skill
  - 新增/修改核心领域模型
  - 跨层调用变更
  - 外部依赖引入
  - 多服务交互流程变更

---

## 核心指令

### 为什么需要Spec RFC

AI编程最大的风险不是技术实现，而是**需求理解偏差**。

模糊输入 → 不可控输出
清晰需求 → 确定性交付

Spec RFC把不确定性从实现阶段前置到需求阶段。

### 五阶段流程

#### 阶段1: 需求启发 (Elicitation)

**目标**: 探索项目已有状态，深度挖掘涉众真实诉求

**AI行为**:
- 主动反问用户，进行信息对齐
- 探索反面情况（退出选项、故障模式）
- 识别与现有功能的交叉点

**发散维度**:
| 维度 | 探索问题 |
|------|---------|
| 意图 | 根本原因是什么？成功标准是什么？ |
| 涉众 | 直接用户是谁？间接用户是谁？ |
| 反面 | 失败场景？取消选项？ |
| 集成 | 与现有功能如何交互？ |
| 完整性 | CRUD？生命周期？ |
| 质量 | 如何验证？可测试性？ |

#### 阶段2: 需求分析 (Analysis)

**目标**: 整理最终需求，按类型分类

**输出格式**:
```yaml
需求类型:
  - 业务需求: ...
  - 用户需求: ...
  - 功能需求: ...
  - 非功能需求: ...
  - 外部接口需求: ...
```

#### 阶段3: 需求定义 (Specification)

**目标**: 撰写需求说明书

**10维度规格**:
1. 背景与目标
2. 需求类型概述
3. 功能需求 (FR-001, FR-002, ...)
4. 非功能需求 (NFR)
5. 外部接口需求
6. 约束与假设
7. 优先级与里程碑
8. 变更/设计提案
9. 待确定事项列表

#### 阶段4: 技术设计 (Technical Design)

**目标**: 从Spec到Code的桥接

**输出内容**:
- 现状分析
- 目标状态
- 设计选项（含优缺点）
- 详细设计
- 实施与迁移计划

#### 阶段5: 验证 (Validation)

**目标**: 保证需求开发产物"足够好"

**8个质量标准检查**:
| 标准 | 检查项 |
|------|--------|
| 完整性 | 所有需求都有对应实现？ |
| 一致性 | 功能间无冲突？ |
| 正确性 | 实现符合Spec？ |
| 可行性 | 技术上可实现？ |
| 必要性 | 每个需求都必要？ |
| 已排序 | 优先级清晰？ |
| 可溯性 | 每个功能可追溯到需求？ |
| 醒豁性 | 无歧义？ |
| 可验性 | 有验收标准？ |

## AI行为准则

### ✅ 应该做的

1. **主动澄清**: 对模糊需求主动反问
2. **输出RFC草稿**: 每个阶段输出结构化文档
3. **用户确认**: 等待用户对RFC满意后再编码
4. **追踪变化**: 需求变更时更新RFC

### ❌ 不要做的

1. **不盲目编码**: 复杂需求未澄清前不直接实现
2. **不假设理解**: 不确定时主动询问
3. **不跳过阶段**: 每个阶段都必须有输出

## 输出模板

### RFC草稿模板

```markdown
# RFC-{序号}-{日期}

## 1. 背景与目标
[一句话描述需求背景和期望达成目标]

## 2. 涉众分析
- 直接用户: [用户描述]
- 间接用户: [用户描述]
- 成功标准: [可量化指标]

## 3. 功能需求
### FR-001: [需求标题]
- 描述: [需求详细描述]
- 输入: [输入数据]
- 输出: [输出数据]
- 验收标准: [如何验证]
### FR-002: ...

## 4. 非功能需求
- 性能: [指标要求]
- 可用性: [指标要求]
- 安全: [指标要求]

## 5. 技术约束
- 架构约束: [约束描述]
- 技术选型: [选型理由]

## 6. 里程碑
- M1: [时间点] - [目标]
- M2: [时间点] - [目标]

## 7. 待确定事项
- [ ] [事项描述]
- [ ] [事项描述]

## 8. 风险评估
- 风险1: [描述] → [缓解措施]
- 风险2: [描述] → [缓解措施]
```

## 何时使用

| 任务复杂度 | 是否使用Spec RFC |
|-----------|-----------------|
| 简单Bug修复 | ❌ 直接修复 |
| 单文件功能添加 | ❌ 直接实现 |
| 跨模块重构 | ✅ 必须执行 |
| 新增外部依赖 | ✅ 必须执行 |
| 多服务交互变更 | ✅ 必须执行 |
| 核心领域模型变更 | ✅ 必须执行 |

**核心原则**: 如果人类开发者需要先"想清楚"才能写代码，这个任务就需要Spec RFC。
