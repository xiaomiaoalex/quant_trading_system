# quant_trading_system 架构定义（Core / Adapter / Control 三平面）

本项目采用 **三平面（Three-Plane）架构**：Core Plane（业务核心）、Adapter Plane（外部 I/O 与容错适配）、Control Plane（治理与运维控制）。三者通过明确的 **Ports/Events 契约** 解耦，确保在高不确定网络环境下仍能保持交易正确性与可演进性。

---

## 1) Core Plane（业务核心，无 I/O）

### 1.1 定位
Core Plane 是系统的“业务真相层”，只表达交易系统的 **领域模型、状态机与决策逻辑**，不依赖任何外部系统（网络/数据库/券商 SDK）。  
核心目标：**交易正确性（Correctness First）** —— 幂等、单调状态机、可回放、可对账。

### 1.2 职责（Must）
- **领域模型（Domain Model）**
  - Instrument / OrderIntent / Order / Execution / Portfolio / Position / PnL
- **订单管理系统 OMS（唯一状态入口）**
  - 严格状态机：`PENDING -> NEW -> SUBMITTED -> PARTIALLY_FILLED -> FILLED/CANCELLED/REJECTED/EXPIRED`
  - 防回滚（Rank 单调）与幂等（cl_ord_id、exec_id/exec_key）
  - Pending 超时（Fail-closed）
- **风控（Risk Engine）**
  - Fail-closed：不确定即拒绝
  - KillSwitch Level 行为约束（L1/L2/L3）
- **事件溯源（Event Sourcing）与回放**
  - canonical event log（append-only）
  - Snapshot + Replay：重启后快速恢复 OMS/Portfolio
- **读模型/投影（可选）**
  - orders/executions/positions/pnl 的投影规则（可由 Adapter/Control 负责执行）

### 1.3 输入/输出契约（Ports & Events）
- 输入：
  - `MarketEvent`（行情）
  - `OrderEvent / ExecutionEvent / BalanceEvent / AdapterHealthEvent`（来自 Adapter 的回报流）
- 输出：
  - `OrderIntent`（策略意图）
  - `OrderRequest`（提交/撤单请求，交给 ExecutionPort）
  - `RiskDecisionEvent / OrderIntentEvent / Snapshot` 等 canonical events

### 1.4 禁止事项（Must Not）
- 禁止网络 I/O、文件 I/O、数据库 I/O、调用券商 SDK
- 禁止依赖真实时间（使用 Clock Port）
- 禁止在多处更新订单状态（必须经 OMS 单入口）

---

## 2) Adapter Plane（外部 I/O + 容错适配层）

### 2.1 定位
Adapter Plane 是系统的“现实世界连接层”，负责所有外部 I/O：WebSocket、REST、券商 SDK、限流、重连、对齐、脏数据缓冲。  
核心目标：**在不稳定网络下输出“可被 Core 消化的干净事件流”**。

### 2.2 职责（Must）
- **市场/券商适配**
  - Public（行情）与 Private（用户/成交）必须 **物理连接与状态机隔离**
  - 统一输出 canonical 消息（Order/Execution/Balance/Health）
- **Dual-Driver Sync（双驱动同步）**
  - WS（主）推进低延迟状态
  - REST（辅）用于纠偏与恢复（重连后强制对齐）
- **Sequence / Version Tracking（乱序/重复/丢失吸收）**
  - 记录 `local_receive_ts_ms`、`exchange_event_ts_ms`、`update_id/seq`
  - 去重（exec_id/exec_key）、乱序标注、gap 触发对齐
- **强制对齐机制（Alignment Gate）**
  - PrivateStream 重连后必须执行一次 **P0 强制 REST Alignment**
  - 对齐成功前不得向 Core 下发 WS 更新（可缓存但不可外发）
- **限流与退避（Rate Limiting & Backoff）**
  - Token Bucket（支持 p0-only 降级）
  - 指数退避（full jitter）+ `Retry-After` 下限
  - 429 时进入“只允许 P0”的降级模式，避免请求风暴
- **网络状态机（Network FSM）**
  - `CONNECTED / DISCONNECTED / RECONNECTING / STALE_DATA / ALIGNING`
  - 静默断流检测必须依赖 `recv timeout` / `pong waiter timeout`（不能靠 TCP keepalive）
- **DEGRADED_MODE 信号**
  - reconnect storm / stale storm / rate-limit storm 进入 DEGRADED
  - 向 Control Plane 发出“环境性风险事件”并触发 KillSwitch L1（详见 Control Plane）

### 2.3 对 Core 的统一接口（Ports）
- `ExecutionPort`
  - `submit(intent)`, `cancel(cl_ord_id)`, `query(cl_ord_id)`
- `ExecutionReportsPort`
  - `stream() -> AsyncIterator[OrderEvent | ExecutionEvent | BalanceEvent | AdapterHealthEvent]`
- 快照查询（辅）
  - `get_open_orders_snapshot()`
  - `get_account_snapshot()`
  - `get_positions_snapshot()`

### 2.4 禁止事项（Must Not）
- 不得实现 OMS/Portfolio 业务规则（只做“事实采集 + 容错整形”）
- 不得输出未标注元数据的回报事件（必须带 source/ts/seq/out_of_order/gap flags）
- 不得在 429/断网时无限重试导致 API 风暴

---

## 3) Control Plane（治理与运维控制层）

### 3.1 定位
Control Plane 是系统的“治理与控制面”，负责策略/风控/熔断/审计/可观测性接口，并把“环境恶化信号”转换为系统级保护动作。  
核心目标：**可运营、可审计、可控、可扩展**。

### 3.2 职责（Must）
- **策略治理（Strategy Studio First）**
  - 策略注册/版本/参数 schema
  - deployment（实例）创建/启停/热更新参数/灰度
- **风控配置与熔断（Risk & KillSwitch）**
  - KillSwitch Level 定义：
    - L0: NORMAL
    - L1: NO_NEW_POSITIONS（禁止新开仓，仅允许平仓/减仓/撤单）
    - L2: CANCEL_ALL_AND_HALT（撤单 + 禁止交易）
    - L3: LIQUIDATE_AND_DISCONNECT（强平 + 断连）
  - 环境风险事件接入：Adapter DEGRADED → 自动触发全局 L1
- **审计与可回放接口**
  - events/snapshots 查询
  - replay 触发（管理员）
- **实时推送**
  - WebSocket streams：订单状态、PnL、AdapterHealth（减少轮询延迟）
- **对账（后续）**
  - 定时拉券商快照，差异触发修复/熔断（以券商为真相）

### 3.3 与 Adapter 的交互（关键）
- Adapter 进入 DEGRADED_MODE 时，Control Plane 必须支持：
  - `POST /v1/risk/events`（dedup_key 幂等）
  - `POST /v1/killswitch`（GLOBAL L1 幂等）
- 幂等要求：
  - dedup_key 必须是确定性窗口哈希（reason + scope + window_start）
  - 上报必须队列化/单 worker、MAX_RETRIES、cooldown 防抖、Fail-closed 本地兜底

### 3.4 禁止事项（Must Not）
- 不得把控制面请求直接阻塞交易执行循环（所有控制动作必须异步/可降级）
- 不得依赖“人工观察”作为熔断触发条件（环境恶化必须自动化）

---

## 4) 三平面交互总览（数据流）

1. **Strategy（Core）** 产生 `OrderIntent`
2. **OMS（Core）** 风控校验 → 生成 `Order`（cl_ord_id）→ 通过 `ExecutionPort` 发送请求
3. **Adapter（WS/REST）** 接收回报/成交 → 做去重/乱序处理/对齐 → 输出 canonical `OrderEvent/ExecutionEvent`
4. **Core（ExecutionReportsPort）** 消费回报事件 → 推进 OMS/Portfolio → 写入 event_log + snapshot
5. **Adapter 进入 DEGRADED** → **Control Plane** 接收风险事件 → 触发 KillSwitch L1 → Core 风控生效

---

## 5) 工程验收口径（关键硬约束）

- **Core**：状态机不可回滚（Rank 单调）；成交不重复；回放可精确重建
- **Adapter**：静默断流可检测；reconnect 单飞；重连后 alignment gate；429 降级 p0-only + Retry-After
- **Control**：环境风险上报幂等；无 API 风暴；Fail-closed 本地锁死必达；flapping 有 cooldown
