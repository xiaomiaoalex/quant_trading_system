## 1. 项目名称

**quant_trading_system _Crypto v3.3.0**

---

## 2. 项目使命

在个人开发者资源约束下，构建一个以 **Binance 为首发交易场所**、以 **趋势 / 量价 / 资金结构 / 链上行为 / 事件驱动** 为核心研究对象、以 **数据深度驱动** 和 **工程确定性** 为首要原则的系统化数字资产交易平台。

该项目不是“自动下单脚本”的工程化包装，也不是模仿大型做市或高频机构的 Quant Cosplay。它的使命是：

- 在加密市场高波动、低信噪比、24/7 连续交易的条件下
- 用统一、可审计、可回放、可治理的基础设施
- 先构建一个 **能活下来的交易系统**
- 再逐步演进为一个 **有研究纪律、有状态识别能力、有策略扩展能力** 的平台

同时，系统在架构上保留未来接入 **中国 A 股券商接口** 的能力，但不在当前版本里提前实现完整 A 股逻辑。

## 4. 项目背景

加密货币交易市场与 A 股、传统期货或股票量化市场不同，具有以下结构性特征：

### 4.1 7x24 连续交易
- 无开盘、无收盘
- 无天然日内结算边界
- 周末和凌晨同样可能发生极端波动
- 风险暴露没有“休市缓冲带”

### 4.2 弱基本面、强叙事
- 大多数交易标的不具备成熟的财务报表体系
- 价值锚点弱
- 价格受叙事、流动性、情绪、资金结构和监管预期影响极大

### 4.3 强趋势、强量价、强衍生品结构影响
- 趋势延续、量价背离、波动压缩与扩张有更强可交易性
- 资金费率、未平仓合约、清算、Basis 等衍生品数据常常携带更强信息密度
- 市场状态切换快，趋势与回撤都更剧烈

### 4.4 链上数据提供独特视角
- 巨鲸移动、交易所流入流出可提前反映潜在抛压或买盘
- 稳定币供应变化反映市场整体风险偏好

### 4.5 数据极脏、执行极脆弱
- WebSocket 静默断流
- REST 与 WS 同步延迟
- 交易所接口限流与错误码风暴
- 插针与局部流动性真空

### 4.6 小资金的优势与约束并存
- 优势：冲击成本低，反应灵活
- 约束：容错率低，无法依赖复杂团队和大规模基础设施兜底

因此，Crypto 版不能套用“季度基本面 + 月频多因子组合”的主线，而必须承认：

> **数字货币版系统的主语，应是“多维数据融合下的状态识别、机制驱动的信号生成、时间与流动性感知的风险控制”。**

## 5. 项目解决的真实问题

### 5.1 数据可信与多样性问题
- Binance WS 是否静默断流
- REST 快照与 WS 增量是否对齐
- 资金费率、OI、清算数据是否连续可信
- 链上数据是否能及时接入并与行情对齐
- 事件公告能否被及时纳入事件流
- 极端脏数据是否会污染信号

### 5.2 工程确定性问题
- 本地订单状态是否可单调推进
- 是否会出现重复记账、逆序状态回滚、重复成交认领
- 是否能在交易所异常时 Fail-Closed
- 是否可回放、可审计、可恢复

### 5.3 研究有效性问题
- 信号是否只是伪相关的量价模式
- 资金费率 / OI / 清算等变量是否真正有增量信息
- 链上行为能否作为早期预警
- 事件驱动是否只是事后解释
- AI 生成的候选规则是否只是噪声

### 5.4 风险生存问题（中低频特有）
- 在插针、暴涨暴跌、断流和交易所异常下是否还能活下来
- 如何应对凌晨、周末的流动性枯竭
- 下单前能否估算滑点，避免深度不足导致巨大损失
- 是否具备断连、未对齐、账户异常时的本地锁死能力
- 是否能控制单币种、单方向、单策略的风险暴露

### 5.5 系统演进问题
- 如何在 Binance 首发前提下，不把系统写死成单交易所脚本
- 如何复用基础设施到未来 A 股券商接入
- 如何让 Research / Portfolio / Execution / AI 共用统一治理框架

## 8. 核心设计原则

### 8.1 数据深度先于策略多样性
在 Crypto 市场中，系统首要问题不是“有没有策略”，而是：
- 行情流是否可信
- 资金费率 / OI / 清算是否完备
- 链上数据能否提供增量信息
- 事件公告是否被结构化

### 8.2 Alignment Gate 先于事件外发
当出现 WS 重连、REST 补快照、序列号跳变、订单薄状态不连续、账户 / 仓位状态可能失真时，系统必须先进入 Alignment Gate。

在未重新对齐前：
- 不外发 Private / Execution 事件
- Public 行情事件可继续外发，但必须附带 `DEGRADED/ALIGNING` 状态标签
- 不允许信号驱动执行
- 必要时触发本地防御动作

### 8.3 状态机必须单调递增，终态不可逆
对订单、成交、仓位等核心对象，必须保证状态 Rank 满足：

`R(S_{t+1}) >= R(S_t)`

要求：
- 终态不可逆
- 不允许状态回滚
- 不允许重复记账
- 不允许相同成交被重复认领

### 8.4 AI 必须 AI-clean 地接入
- Core Plane 必须 AI-clean
- AI 只能在 Insight / Research / Report 层活动
- AI 可以输出 `AIInsightEvent`
- AI 不能直接穿透到 Core 执行下单

### 8.5 机制先于回测
任何候选策略必须附带市场微观结构假设，解释其盈利来源及失效条件。回测仅用于验证，不应用来神化伪规律。

### 8.6 时间与流动性风险前置
针对中低频持仓，必须在风控中加入：
- 时间段系数（如凌晨、周末降低仓位上限）
- 订单簿深度检查（下单前估算滑点，超过阈值则拒绝或拆单）
- 极端行情逃生时间模拟

### 8.7 小资金优先“活下来”
当前版本不追求复杂资产配置，而追求：
- 仓位边界
- 风险上限
- 成本透明
- 执行稳定
- 极端行情生存

### 8.8 多市场适配靠抽象，不靠提前实现
当前不实现 A 股交易逻辑，但必须在接口层保证：
- 不把 Binance 写死到业务层
- 订单、账户、行情、符号元数据都通过抽象接口暴露
- 未来中国券商接口可以复用 Core / Persistence / Policy / Audit 基础设施

### 8.9 复用商品化部件，自研差异化骨干
以下层优先复用：
- 交易所基础接入
- 基础网络与重试机制
- 指标库与数值计算
- 基础监控与日志
- 部分研究 / 回测底座

以下层坚持自研：
- Core 状态机与幂等层
- Alignment Gate
- Policy Plane
- Feature Store 版本治理
- 审计 / 回放 / Reconciler / Risk 收口
- AI 治理边界

## 9. 数据源策略（Crypto 版）

### 9.1 Layer A：高频确定性底盘（交易所直连层）
当前来源：
- Binance 直连（REST + WebSocket）
- 可基于 CCXT / CCXT Pro 做统一封装，但业务层不得直接依赖具体 SDK 细节

核心字段：
- Kline / OHLCV
- Trades / AggTrades
- Orderbook / Depth
- Ticker / Mark Price
- Funding Rate
- Symbol Metadata
- Exchange Status
- Account / Balance / Position / Open Orders

关键风险：
- WS 假死
- 增量深度丢包
- REST / WS 不一致
- 429 限流风暴
- 交易所状态异常

系统要求：
- 必须建立 `Alignment Gate`
- 必须有重连单飞控制
- 必须有快照重建机制
- 必须有延迟与异常状态检测
- Reconciler 为主闭环增强能力

### 9.2 Layer B：衍生品与链上辅助层（P1）
必须接入字段：
- Funding Rate（实时与历史）
- Open Interest（实时与历史）
- Large Liquidations（实时）
- Exchange Netflows
- Stablecoin Supply Ratio
- Long / Short Ratio

### 9.3 Layer C：非结构化叙事与事件层（P1）
必须接入来源：
- Binance 官方公告
- 主要项目方公告
- 监管机构公告
- Crypto News RSS（过滤后）

原则：
1. 交易所和项目官方原文
2. 监管和法定公告
3. 媒体聚合和 RSS
4. 社交情绪仅作辅助

## 10. 五平面技术架构（概要）

### 10.1 Core Plane
职责：订单状态机、持仓状态机、成交归并、幂等、领域事件生成。

### 10.2 Adapter Plane
职责：连接外部世界、吸收脏数据、实现对齐、标准化事件，并隔离第三方依赖。

### 10.3 Persistence Plane
职责：存储 event_log、交易对象、原始数据、特征、风险事件、研究工件和 AI 输出。

### 10.4 Policy Plane
职责：时间窗口、深度检查、杠杆限制、风险预算、KillSwitch、运行模式控制。

### 10.5 Insight Plane
职责：研究、信号、实验、AI 摘要与解释，不直接执行下单。

## 11. 最小可生存闭环定义（v3.3.0）

当前版本的最小可生存闭环定义为：

`Binance data`
+ `Funding / OI data`
+ `Event data`
+ `基础链上 / 宏观数据`
-> `health check`
-> `alignment gate`
-> `standardized events`
-> `feature store`
-> `core state machine`
-> `time-aware & depth-aware risk gate`
-> `minimal signal`
-> `target position`
-> `order intent / simulated or guarded execution`
-> `reconcile`
-> `real-time monitoring alert`
-> `report / replay`

只要这条链路成立，系统就具备在真实市场中以中低频策略生存和迭代的资格。

## 12. 当前阶段的禁止事项

- 先写复杂策略，后补数据接入
- 先写自动执行，后补 Reconciler 与监控
- 先做 AI 自动化，后补风控
- 先做多市场扩展，后补 Binance 主链路
- 先做华丽面板，后补事件溯源
- 把第三方交易 SDK 直接写进 Core
- 在 Feature Store 未就绪前，直接硬编码特征计算到策略中
- 忽视时间窗口和流动性检查，直接下单
- 把 Future Equity Domain 写成当前版本的大型实现工程
