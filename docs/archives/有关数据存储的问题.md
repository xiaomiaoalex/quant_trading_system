我看下来：**有问题，而且不是单点 bug，是存储职责边界还没收口**。项目已经有 PG event store、executions 表、position_lots、projection、snapshot 等基础，但控制面服务仍大量默认走全局内存存储，导致“内存事实源”和“PG 事实源”并存。

**主要风险**

1. **内存无限增长**
   [in_memory.py](<D:\Code of Python\quant_trading_system\trader\storage\in_memory.py:60>) 里 `executions`、`execution_by_key`、`events`、`snapshots`、`strategy_runtime_states` 等都是常驻结构；`append_event()` 只追加，[list_events()](<D:\Code of Python\quant_trading_system\trader\storage\in_memory.py:671>) 还会先复制全量列表再过滤。运行久了会同时有内存和查询性能问题。

2. **高价值数据仍可能只在内存**
   `OrderService`、`PortfolioService`、`KillSwitchService`、`ReconcilerService` 默认都用 `get_storage()`，也就是控制面内存存储。订单查询、成交查询、持仓视图、KillSwitch 状态、对账漂移事件都有重启丢失风险。

3. **事件存储语义混乱**
   [EventService](<D:\Code of Python\quant_trading_system\trader\services\event.py:21>) 明确写着 events “always in-memory”。这和项目的 event sourcing 目标冲突。更细一点，`EventService._to_stream_events()` 读 `payload`，而 `ControlPlaneInMemoryStorage.append_event()` 存的是 `data`，回放时 payload 可能为空。

4. **deployment runtime 状态仍有 ID 维度隐患**
   runner 已经区分 `runtime_id = deployment_id or strategy_id`，但 [save_strategy_runtime_state](<D:\Code of Python\quant_trading_system\trader\storage\in_memory.py:745>) 仍按 `strategy_id` 存。多个 deployment 使用同一个 strategy template 时会互相覆盖。

5. **PG 能力存在但没有统一接入**
   PG 里已有 `executions` 唯一约束、`position_lots`、`strategy_positions_proj`、`reconciliation_log` 等迁移；但 OMS callback 现在仍主要写 `self._storage.create_execution()` 和 `upsert_position()`，也就是当前服务注入什么就写什么，没有强制 PG-first。

**我建议的设计边界**

把数据分三类：

- **必须持久化，不能回退成内存事实源**：订单状态流、成交 fills、`cl_ord_id + exec_id` 去重记录、持仓/lot、KillSwitch 变更、risk upgrade/effect、部署生命周期、runtime state、对账日志、策略代码/版本/参数。
- **可重建的读模型，但生产也应持久化**：orders projection、strategy/account positions projection、PnL、monitor snapshot、前端列表页数据。
- **只允许内存缓存**：SSE 连接、最近 UI 事件窗口、短 TTL dedup cache、crawler recent ids、replay job 临时状态、健康检查采样。

**落地方案**

第一步先做止血：给 `ControlPlaneInMemoryStorage` 所有 append-only 结构加边界。`events` 改 `deque(maxlen=N)` 或按 stream 分桶，`snapshots` 每个 stream 只留最新 N 个，`execution_by_key` 用 TTL/LRU，replay jobs 和 crawler `_processed_ids` 也加 TTL。并加指标：当前 event 数、execution 数、snapshot stream 数、被丢弃的非关键事件数。

第二步把 P0 数据改成 PG-first：成交写 PG `executions`，事件写 PG `event_log`，lot/持仓写 `position_lots` 和 projector，KillSwitch/risk 写 PG repository。交易关键写入失败时不要静默 fallback 到内存，应该进入 degraded/fail-closed。

第三步统一服务入口：`OrderService`、`PortfolioService`、`KillSwitchService`、`EventService` 不要直接默认全局 `get_storage()`，改成 repository/interface 注入。内存实现只用于测试和 dev fallback，生产读写走 PG repository。

第四步修正 deployment 维度：runtime state、events、orders/executions、positions 的运行实例维度应优先用 `deployment_id`，`strategy_id` 只表示策略模板。runtime state 表或 key 应改为 `deployment_id` 主键，并保留 `strategy_id` 字段用于归因。

一句话总结：**当前内存存储可以当 demo/dev cache，但不适合作为生产事实源。最优解不是“把内存调大”，而是把内存降级为有 TTL/上限的 cache，把订单、成交、持仓、风险、部署运行态统一收敛到 PG event log + projector + snapshot。**
