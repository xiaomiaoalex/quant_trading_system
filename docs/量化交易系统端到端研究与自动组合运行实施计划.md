# 量化交易系统端到端研究与自动组合运行实施计划

## Summary

目标按你的选择收敛为：**Crypto 核心数据源优先、研究平台优先、最终支持全自动策略组合启停/降仓/恢复**。第一阶段不急着真钱实盘，而是先把“数据 -> 分析 -> 策略开发 -> 回测 -> 门禁 -> 自动组合决策 -> paper/shadow 运行 -> 监控审计”做成一条稳定流水线；live 模式只在同一套链路验证完后打开。

当前仓库已有策略部署、回测、FeatureStore、StrategyLifecycleManager、CapitalAllocator、RiskSizer、portfolio-research API 等基础。实施重点不是重写，而是把这些能力串成统一生命周期，并修复前端 Strategy Lab 当前的契约断点。

## Key Changes

### 1. 先修通 Strategy Lab 最小闭环

- 修复 `Frontend/src/pages/Backtests.tsx`：
  - `loadStrategy` 必须传 `symbols`、`account_id`、`venue`、`mode`、`deployment_id`。
  - `startStrategy` / `stopStrategy` 必须使用 `deployment_id`，不能再用 `strategy_id`。
  - 页面表单新增：symbols、account_id、venue、mode、deployment_id。
  - 加载成功后保存返回的 `deployment_id`，后续启动、停止、事件查看都以它为准。
- 前端按钮顺序固定为：
  - Debug Code
  - Save Version
  - Run Backtest
  - Promote / Load Deployment
  - Start Paper/Shadow
  - Stop
- 后端不新增复杂能力，只补齐必要的错误提示：
  - 未传部署参数时返回清晰 422。
  - 策略未通过回测门禁时，Promote 接口拒绝部署。
- 验收标准：
  - `npm run typecheck` 通过。
  - 一个示例策略能从前端保存、调试、回测、加载、启动、停止。
  - 所有运行动作都能在 deployment events 中查到。

### 2. 建立统一策略生命周期工作流

- 新增或收敛为一个后端主流程：`StrategyCandidate`。
- 生命周期状态固定为：
  - `DRAFT`
  - `DEBUG_PASSED`
  - `BACKTEST_RUNNING`
  - `BACKTEST_PASSED`
  - `VALIDATION_PASSED`
  - `APPROVED_FOR_PAPER`
  - `PAPER_RUNNING`
  - `PAUSED_BY_RISK`
  - `STOPPED`
  - `REJECTED`
- 前端 Backtests/Strategy Lab 不再直接拼散接口，而是围绕 candidate 展示状态。
- 使用现有 `StrategyLifecycleManager` 作为状态机基础；缺字段时先更新 `docs/INTERFACE_CONTRACTS.md`。
- 每次状态迁移都要写审计事件：
  - 操作者或系统组件
  - 原状态
  - 新状态
  - 原因
  - 关联 code_version / backtest_run_id / deployment_id
- 验收标准：
  - 用户能看到一个策略从草稿到 paper 运行的完整状态链。
  - 非法跳转被拒绝，例如未回测不能部署。
  - 重复点击不会创建重复部署。

### 3. 把回测升级为研究平台核心

- 回测请求必须显式选择：
  - symbol 列表
  - 时间区间
  - feature_version
  - 初始资金
  - 手续费模型
  - 滑点模型
  - benchmark
- 第一版数据范围只支持 Crypto 核心源：
  - Binance OHLCV / trades / orderbook 可用数据
  - funding / OI
  - 已有 onchain adapter 能提供的数据
  - announcements / event regime
- `FeatureStore` 成为回测数据读取入口；没有真实数据时要显示 `missing_data`，不能静默使用 synthetic 数据冒充真实回测。
- synthetic deterministic backtest 保留，但只标记为 `dev_smoke`，不能作为部署准入依据。
- 回测报告必须输出：
  - return
  - max_drawdown
  - sharpe / sortino
  - win_rate
  - turnover
  - fee/slippage impact
  - exposure by symbol
  - data_quality_summary
  - pass/fail decision
- 验收标准：
  - 前端能选择数据集和 feature_version。
  - 回测报告能区分真实研究回测和 dev_smoke。
  - 只有真实研究回测通过后才能进入 validation gate。

### 4. 接入策略准入门禁

- 使用现有策略验证门禁作为部署前必经步骤。
- 第一版门禁规则固定为：
  - 代码 debug 通过。
  - 真实回测通过。
  - max_drawdown 不超过配置阈值。
  - 数据质量达到阈值。
  - 成本压力测试后仍为正期望。
  - OOS 区间不崩溃。
- 门禁结果写入 candidate：
  - `passed`
  - `failed_rules`
  - `metrics`
  - `evidence_refs`
- 前端显示门禁结果，但不允许用户手动绕过。
- 验收标准：
  - 失败策略无法 Promote。
  - 门禁失败原因在 UI 中可读。
  - 测试覆盖每个门禁失败分支。

### 5. 建立仓位分配与组合预算系统

- 把 `CapitalAllocator` 和 `RiskSizer` 接到策略信号到 OMS 之间。
- 统一交易链路改为：
  - Strategy signal
  - RiskSizer 计算建议仓位
  - CapitalAllocator 按组合预算裁剪
  - Pre-trade risk gate
  - OMS
  - Broker adapter
- 新增策略预算配置：
  - strategy_id
  - deployment_id
  - max_notional
  - max_symbol_exposure
  - max_portfolio_weight
  - min_confidence
  - allow_short
  - priority
  - enabled
- 每个信号必须记录：
  - raw_requested_size
  - risk_sized_qty
  - allocated_qty
  - final_order_qty
  - allocation_decision
  - reject_or_clip_reason
- 前端新增 Portfolio Allocation 页面：
  - 按策略查看预算、当前占用、剩余额度、最近裁剪/拒绝原因。
  - 支持启用/禁用策略预算，但 live 生效前必须走审计。
- 验收标准：
  - 策略不能绕过分配器直接下单。
  - 超预算信号被裁剪或拒绝。
  - 多策略同向暴露超过阈值时，低优先级策略先被裁剪。

### 6. 建立全自动组合控制器

- 新增 `PortfolioRuntimeController`，第一版运行在 paper/shadow，可配置切到 live。
- 控制器定时读取：
  - 每个 deployment 状态
  - 策略实时 PnL / drawdown
  - 组合总暴露
  - symbol 暴露
  - 数据新鲜度
  - FeatureStore 数据质量
  - KillSwitch 状态
  - 最近策略错误事件
  - 回测与 live 表现偏离
- 控制器输出动作：
  - `START`
  - `PAUSE`
  - `RESUME`
  - `STOP`
  - `REDUCE_ALLOCATION`
  - `DISABLE_ALLOCATION`
- 第一版自动规则固定：
  - 全局 KillSwitch 非 NORMAL：停止或暂停所有策略。
  - 数据 stale：暂停依赖该数据源的策略。
  - 策略 drawdown 超阈值：暂停该策略。
  - 组合暴露超阈值：按 priority 降仓。
  - 策略连续错误超阈值：停止该 deployment。
  - 表现恢复且风险正常：允许自动恢复 paper/shadow 策略。
- 所有自动动作必须写审计事件，并能在 Audit/Replay 中回放。
- 验收标准：
  - 构造风险超限场景时，控制器能自动暂停或降仓。
  - 风险恢复后，符合规则的策略能自动恢复。
  - 每个动作都有可解释原因和输入快照。

### 7. 前端产品化工作台

- 导航新增：
  - Data
  - Research
  - Strategy Lab
  - Backtests
  - Portfolio Allocation
  - Portfolio Autopilot
  - Monitor
  - Audit / Replay
- Data 页面显示：
  - 数据源状态
  - symbol 覆盖
  - 最新时间戳
  - feature_version
  - 数据质量
- Research 页面接入现有 `/api/portfolio-research/*`：
  - 运行组合研究
  - 查看候选策略
  - 查看 committee 结果
  - 将候选策略生成 StrategyCandidate
- Strategy Lab 聚焦代码与参数。
- Backtests 聚焦历史验证和报告。
- Portfolio Autopilot 聚焦自动启停/降仓决策。
- 验收标准：
  - 用户不用跳接口工具，就能完成完整研究到运行流程。
  - 每个页面只负责一个阶段，状态通过 candidate/deployment 串联。

## Public Interfaces / Types

- 新增或统一后端 DTO：
  - `StrategyCandidate`
  - `StrategyCandidateStatus`
  - `BacktestDatasetSpec`
  - `BacktestGateResult`
  - `StrategyAllocationProfile`
  - `AllocationTrace`
  - `PortfolioAutopilotDecision`
  - `PortfolioAutopilotSnapshot`
- 新增或收敛 API：
  - `POST /v1/strategy-candidates`
  - `GET /v1/strategy-candidates/{candidate_id}`
  - `POST /v1/strategy-candidates/{candidate_id}/debug`
  - `POST /v1/strategy-candidates/{candidate_id}/backtests`
  - `POST /v1/strategy-candidates/{candidate_id}/validate`
  - `POST /v1/strategy-candidates/{candidate_id}/promote`
  - `GET /v1/allocations`
  - `PUT /v1/allocations/{deployment_id}`
  - `GET /v1/allocations/{deployment_id}/traces`
  - `GET /v1/portfolio-autopilot/snapshot`
  - `POST /v1/portfolio-autopilot/tick`
  - `GET /v1/portfolio-autopilot/decisions`
- 命名约束：
  - `strategy_id` 只代表策略模板。
  - `deployment_id` 只代表运行实例。
  - `candidate_id` 代表研究/开发生命周期实体。
  - `feature_version` 必须贯穿数据、回测、验证和部署审计。
- 持久化默认：
  - candidate、backtest result、validation result、allocation profile、autopilot decision 必须持久化。
  - 现有 in-memory 可保留 dev fallback，但不能作为生产默认。

## Test Plan

- 前端：
  - `npm run typecheck`
  - Strategy Lab 表单契约测试。
  - candidate 状态流 UI 测试。
  - allocation 页面渲染和更新测试。
- 后端单测：
  - candidate 状态机合法/非法迁移。
  - backtest dataset 缺失时 fail-closed。
  - validation gate 每条规则失败场景。
  - CapitalAllocator/RiskSizer 接入后 qty 裁剪链路。
  - PortfolioRuntimeController 的 start/pause/resume/stop/reduce 决策。
- 集成测试：
  - 策略代码保存 -> debug -> 真实回测 -> validate -> promote -> start paper。
  - 多策略同时发信号时，组合预算正确裁剪。
  - 数据 stale 导致自动暂停。
  - KillSwitch 升级导致全策略暂停。
  - 审计事件可 replay。
- 回归测试：
  - 现有 P0 回归集必须继续通过。
  - 策略部署旧接口如果保留，必须有兼容测试。
  - synthetic dev_smoke 回测不能误标为部署准入通过。

## Assumptions

- 第一阶段以 **研究平台 + paper/shadow 自动组合运行** 为目标，不直接默认开启真钱 live。
- 数据源第一阶段限定为 **Crypto 核心源**，暂不强依赖 Coinglass、Glassnode 等付费数据。
- 你选择了 **全自动组合**，所以计划会实现自动启停/降仓/恢复；但上线顺序先在 paper/shadow 验证，再通过配置开放 live。
- 所有接口、DTO、跨层调用变更必须先更新 `docs/INTERFACE_CONTRACTS.md`；涉及架构边界和数据流变化必须更新 `docs/PROJECT_ARCHITECTURE.md`。
- 每个阶段都按 TDD：先写基于真实现有接口的失败测试，再做最小实现，最后更新 `PROJECT_STATUS.md`、`DEVELOPMENT_LOG.md` 和 `docs/EXPERIENCE_SUMMARY.md`。
