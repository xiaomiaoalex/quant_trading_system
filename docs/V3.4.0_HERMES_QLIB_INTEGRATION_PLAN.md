# quant_trading_system Crypto v3.4.0 开发计划（Hermes + Qlib 接入）

> 目标：在不破坏现有五平面与确定性约束的前提下，把 Qlib 研究能力与 Hermes 编排能力接入系统，形成“研究可复现、执行可审计、上线可治理”的 AI 量化闭环。

---

## 1. 约束与原则

1. Qlib 只进入 Insight/Research 路径，不直接触发下单。
2. Hermes 只做研发编排和任务调度，不进入 Core/Policy 执行链路。
3. Core Plane 保持 IO-clean / AI-clean，执行依旧走 StrategyRunner + RiskEngine + OMS。
4. 所有新能力必须具备版本化（feature_version/model_version/signal_version）和审计链路。
5. 未通过门控（回测合规、样本外、成本压测、影子验证）的策略禁止进入 RUNNING。

---

## 2. 分阶段落地顺序

### Phase A（P0）研究数据标准化与契约冻结

**目标**：先统一数据语义，再接入模型，避免“训练与实盘两套宇宙”。

**交付物**：
- `scripts/qlib_data_converter.py`（或等价模块）：将 Feature Store / K线数据转换为 Qlib 可训练格式。
- `docs/` 中新增数据契约：字段、时间戳、时区、复权/连续合约规则、缺失值策略。
- `feature_version` 映射规范：训练集与线上推理必须可追踪到同一版本簇。

**验收标准**：
- 同一 symbol + 时间窗，转换结果可重复生成（确定性）。
- 训练输入可回放到源数据（event/raw snapshot reference）。
- 缺失数据、异常跳点、对齐失败触发 Fail-Closed 标记。

### Phase B（P0）Qlib 离线训练与模型注册

**目标**：形成稳定离线训练流水线，不改动线上执行路径。

**交付物**：
- `scripts/qlib_train_workflow.py`：训练、验证、导出预测。
- `scripts/qlib_factor_miner.py`：因子重要性挖掘（先 LightGBM）。
- 模型注册清单：`model_version`, `feature_version`, `train_window`, `label_def`。

**验收标准**：
- 任意一次训练可重跑并复现关键指标。
- 指标报告包含基线策略对照（非 AI baseline）。
- 产物（模型、报告、参数）可审计可追溯。

### Phase C（P1）Qlib 信号桥接到 Strategy 协议

**目标**：把 Qlib 预测映射为系统标准 Signal，不绕过风控。

**交付物**：
- `scripts/qlib_to_strategy_bridge.py`：预测值 -> `Signal`（含置信度、理由、版本标签）。
- 桥接策略模板：只在 Insight/Service 生成信号，执行仍由 Runner 驱动。
- 信号门控：最小交易阈值、冷却期、方向一致性检查。

**验收标准**：
- 每条 AI 信号都带 `model_version/feature_version/trace_id`。
- Signal 进入 RiskEngine 后行为可解释（PASS/REDUCE/REJECT/HALT）。
- 失败路径（模型缺失/特征缺失）默认拒绝交易。

### Phase D（P1）Hermes 编排接入（研发侧）

**目标**：用 Hermes 提升研究效率，不侵入交易运行时。

**交付物**：
- `insight` 侧任务模板：数据准备、训练、回测、报告生成 SOP。
- Skill/脚本化编排：一键运行“数据转换->训练->评估->导出信号草案”。
- 审计记录：谁触发、用了什么配置、产出哪些 artifacts。

**验收标准**：
- Hermes 只调用研究脚本，不直接调用下单接口。
- 编排失败可重试，可定位到具体步骤。
- 输出统一落盘到报告/审计存储。

### Phase E（P1）上线前五层门控 + 影子验证

**目标**：避免“回测很好看，上线就失真”。

**交付物**：
- 接入现有 `strategy_validation_gate`：机制假设、回测合规、样本外、成本压测、影子模式。
- AI 策略专用报表：基线对比、成本敏感性、风控干预率。
- 影子模式对比：回测信号 vs 影子信号 vs 实际成交偏差。

**验收标准**：
- 1.5x 成本下仍正期望才可审批。
- 影子偏差超阈值自动告警并阻断上线。
- 审批流保留完整 HITL 记录。

### Phase F（P2）生产收敛与运维观测

**目标**：把 v3.4.0 能力变成长期可维护资产。

**交付物**：
- 运行面板：模型版本、特征新鲜度、信号触发率、风控干预率。
- 定期回归任务：模型漂移检测、数据质量巡检。
- 回滚策略：模型级回滚、信号级降级、KillSwitch 联动。

**验收标准**：
- 模型/信号异常可在分钟级发现并隔离。
- 回滚不影响 Core 状态一致性。
- 文档与状态页保持同步更新。

---

## 3. 工作分解（建议 4 周节奏）

- Week 1：Phase A + B（数据契约、转换器、首版训练流水线）
- Week 2：Phase C（信号桥接、版本标签、风险门控联调）
- Week 3：Phase D + E（Hermes 编排、五层门控、影子验证）
- Week 4：Phase F（观测、回滚、文档收敛、上线演练）

---

## 4. v3.4.0 DoD（Definition of Done）

1. Qlib 已可稳定产出版本化预测信号。
2. Hermes 已能稳定编排研究工作流，且不越权到执行链路。
3. AI 信号通过统一桥接进入现有 StrategyRunner/RiskEngine。
4. 五层验证门控 + HITL 审批可阻断不合格策略。
5. 文档、状态、计划三者一致，无版本漂移。
