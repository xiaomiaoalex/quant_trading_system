## 0. 执行总则

1. 规范优先：一切实施以 v3.0.5 为唯一执行规范。  
2. 三态推进：严格按 `As-Is -> In-Progress -> Target` 演进，不跨级“跳实现”。  
3. 先稳后快：优先完成 P1 收口（Task10.3 + 契约测试 + CI Gate），再进入 P2 主线。  
4. AI 边界：AI 只做大脑与治理，不进入 Core 确定性执行路径。  
5. 门禁驱动：每个 Sprint 以可验证 Gate 作为 Done 判据。

---

## 1. 蓝图（Blueprint）

## 1.1 目标态蓝图

系统分 5 个工作平面同步演进：

1. Core Plane：状态机单调、幂等、Fail-Closed。  
2. Adapter Plane：Public/Private 隔离、重连、对齐、退避、降级级联。  
3. Persistence Plane：PostgreSQL 事件溯源 + 风险幂等持久化 + 可回放。  
4. Policy Plane：KillSwitch 分级治理、规则可审计。  
5. Insight Plane：观测、复盘、AI 提案（仅建议，不执行）。

## 1.2 当前基线（As-Is）

1. P0 主线稳定（连接器、私有流、确定性层、风险闭环、健康检查三级化）。  
2. PostgreSQL 已有 `event_log/snapshots` 基础实现。  
3. `risk_events/risk_upgrades` 尚未落库，重启后幂等契约仍在 In-Progress。  
4. AI 战略与边界已定义，工程能力尚未落地为 API + Gate。

## 1.3 本计划成功标准

1. P1 收口完成：Task10.3 全部交付，`postgres-integration` 强制通过。  
2. P2 主线启动：Runner/Reconciler/Observability 至少形成 MVP 闭环。  
3. AI 治理 MVP：proposal + approve（HITL）与离线评测机制跑通。  
4. 代码、测试、规范三者一致，不出现“文档能力 > 代码能力”漂移。

---

## 2. 项目分轨（Workstreams）

为避免单线瓶颈，按 5 条工作流并行推进：

1. WS-A（可靠性与核心一致性）  
- 状态机、幂等、Fail-Closed、时间体系收口。  

2. WS-B（持久化与回放）  
- Task10.3、事务语义、故障恢复、重启后幂等。  

3. WS-C（控制面与治理）  
- 风险接口契约、KillSwitch、健康检查、运维接口。  

4. WS-D（可观测与工程效率）  
- OTel/Prometheus、迁移管理、测试容器化、CI 强化。  

5. WS-E（AI 治理与策略辅助）  
- AI 提案链路、审批链路、评测门禁、影子运行。

## 3. Sprint 节奏与排期

建议采用 1 周一个 Sprint（个人开发者 + AI 辅助最匹配），先排 10 个 Sprint。

| Sprint | 主题 | 目标 | 核心交付 | Gate / DoD |
|---|---|---|---|---|
| Sprint 0 | 启动与冻结 | 冻结实施口径与验收定义 | 路线图确认、任务拆解、风险登记表 | 计划评审通过 |
| Sprint 1 | Task10.3-A | 风险事件持久化 | `risk_events` 表 + repository 接口 | 单测通过，接口不回归 |
| Sprint 2 | Task10.3-B | 升级幂等持久化 | `risk_upgrades` 表 + upgrade key 唯一约束 | 重复升级无副作用 |
| Sprint 3 | Task10.3-C | 原子语义与恢复 | 事务边界 + 故障补偿策略实现 | 故障注入测试通过 |
| Sprint 4 | Contract Gate | 重启后幂等契约化 | `test_risk_idempotency_persistence.py` + CI 接入 | `postgres-integration` 强制通过 |
| Sprint 5 | Observability MVP | 可观测闭环 | OTel tracing + Prom 指标最小集 | 关键路径指标可见 |
| Sprint 6 | Reconciler MVP | 对账骨架落地 | Candidate/Confirmed 分类 + 手动触发 | 核心流程联测通过 |
| Sprint 7 | Runner MVP | 策略运行闭环 | Registry + Runner 最小联动 | 端到端 dry-run 通过 |
| Sprint 8 | AI Governance MVP | AI 提案不执行链路 | `/v1/ai/proposals` + `/approve` + 审计字段 | HITL 必经、不可绕过 |
| Sprint 9 | AI Eval Gate | AI 评测门禁上线 | 离线评测任务 + 阈值阻断 | 未达阈值禁止灰度 |
| Sprint 10 | 稳定化发布 | 收口与发布候选 | 文档/测试/运维脚本收口 | 发布清单完成 |

## 4. Sprint 详细计划（执行版）

## 4.1 Sprint 0（启动）

1. 输出本计划并冻结优先级。  
2. 建立任务看板（Backlog -> In Progress -> Review -> Done）。  
3. 固定每周评审节奏与状态汇报模板。  

验收：
1. 范围冻结。  
2. 风险与依赖清单可见。  
3. 每个 Sprint 都有 Gate 指标。

## 4.2 Sprint 1-4（P1 收口主线）

### Sprint 1
1. 在 PostgreSQL 层新增 `risk_events` 结构与访问接口。  
2. 将 `ingest_event` 从内存路径桥接到持久层。  
3. 保持 `/v1/risk/events` 的 201/409 语义不变。  

### Sprint 2
1. 新增 `risk_upgrades` 结构与访问接口。  
2. 升级流程写入唯一 `upgrade_key`。  
3. 并发重复请求不触发二次升级。

### Sprint 3
1. 实现 Task10.3 事务时序（BEGIN -> 去重 -> 升级记录 -> 副作用 -> COMMIT）。  
2. 实现故障恢复策略（至少覆盖 Step2/Step6/Step8 断点）。  
3. 增加恢复日志与审计字段。

### Sprint 4
1. 补齐“重启后幂等”契约测试。  
2. 将契约测试接入 `postgres-integration`。  
3. 失败即阻断 PR 合并。

阶段验收（P1 收口完成）：
1. 重启后重复 `dedup_key` 返回 409。  
2. 重启后重复 `upgrade_key` 不产生重复升级副作用。  
3. CI 全绿且关键用例不可跳过。

## 4.3 Sprint 5-7（P2 主线起步）

### Sprint 5（可观测）
1. 引入 OTel tracing（API -> service -> storage 主链路）。  
2. 引入 Prometheus 指标（风险事件、升级动作、重试、故障恢复计数）。  
3. 输出最小运维看板指标字典。

### Sprint 6（Reconciler）
1. 实现对账实体与差异分类（Candidate/Confirmed）。  
2. 支持手动触发与结果查询。  
3. 形成最小告警策略（仅记录，不自动执行高风险动作）。

### Sprint 7（Runner）
1. Strategy Registry 与 Runner 联动最小闭环。  
2. 支持策略参数版本读取。  
3. 支持 dry-run 和 replay 模式。

阶段验收：
1. Runner 可执行最小策略循环。  
2. Reconciler 能输出可解释差异分类。  
3. 关键链路有 trace + metrics。

## 4.4 Sprint 8-10（AI 治理与发布）

### Sprint 8（AI 提案链路）
1. 新增 `AIInsightEvent`、`AIDecisionProposal` 数据模型。  
2. 新增 proposal 创建与审批接口。  
3. 审批前后全链路审计字段落库。

### Sprint 9（AI 评测门禁）
1. 建立离线评测任务（固定数据集 + rubric）。  
2. 配置阈值（结构化输出通过率、误报漏报指标）。  
3. 未达阈值禁止灰度发布。

### Sprint 10（发布收口）
1. 文档、代码、测试三对齐。  
2. 更新 runbook 与发布检查清单。  
3. 形成 v3.0.5 阶段性发布候选。

## 6. 验收标准（Definition of Done）

每个 Sprint 必须同时满足：

1. 功能 DoD：需求范围内功能可运行且行为符合规范。  
2. 测试 DoD：新增/变更代码有对应单测或集成测试。  
3. 契约 DoD：接口语义（状态码、字段、幂等）无破坏性漂移。  
4. 文档 DoD：更新 v3.0.5 Spec 的状态与 Changelog。  
5. CI DoD：相关 Gate 全绿。

---

## 7. 风险清单与对策

1. 风险：Task10.3 事务设计复杂，易出现“部分成功”。  
- 对策：先实现最小原子时序，再做故障注入验证。  

2. 风险：规范与代码继续漂移。  
- 对策：每 Sprint 结束强制执行 Spec Changelog 更新。  

3. 风险：AI 能力过早耦合执行链。  
- 对策：强制 HITL，审批链路前禁止执行。  

4. 风险：个人开发者并行过多导致交付延迟。  
- 对策：每 Sprint 只设 1 个主目标 + 1 个次目标。
