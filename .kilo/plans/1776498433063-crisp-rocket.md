# Task 11-15 开发计划：策略自动交易闭环

## 1. 概述

本计划实现"策略自动交易闭环"，使策略从代码注册→加载→启动→实时行情→信号→真实下单→成交回调全链路真实可运行。

**前提条件**：系统已有 `StrategyRunner.tick()`、Binance 公有流适配器、 OMS/券商接口。

---

## 2. 任务拆分

### Task 11：实时行情订阅 + `runner.tick()` 调度

**目标**：`POST /v1/strategies/{id}/start` 后策略持续收到实时行情并触发 `runner.tick()`

**实施步骤**：

| 步骤 | 操作 | 文件 |
|------|------|------|
| 2.1 | 创建 `StrategyRuntimeOrchestrator` 服务 | `trader/services/strategy_runtime_orchestrator.py` |
| 2.2 | 定义 `RuntimeContext` dataclass（包含 task/queue/symbol/status） | 同上 |
| 2.3 | 实现 `start_strategy(strategy_id)`：订阅行情 + 创建 asyncio.Task 循环调用 tick | 同上 |
| 2.4 | 实现 `stop_strategy(strategy_id)`：取消订阅 + 取消 Task | 同上 |
| 2.5 | 实现 `unload_strategy(strategy_id)`：清理所有资源 | 同上 |
| 2.6 | 在 API routes 的 `/start` 端点调用 orchestrator.start_strategy() | `trader/api/routes/strategies.py` |
| 2.7 | 在 `/stop`、`/unload` 端点调用对应停止/清理方法 | 同上 |
| 2.8 | 与 Binance 公有流适配器对接，实现 `on_market_data` 回调 | `trader/adapters/stream/binance_public_stream.py` |
| 2.9 | 添加运行事件发布（tick、异常、停止原因） | `strategy_runtime_orchestrator.py` |

**验收标准**：
- 启动后 `tick_count` 持续增长
- 停止后 `tick_count` 不再增长
- 多策略并发互不影响

**风险点**：
- Binance 连接断开处理：实现自动重连 + REST 对齐
- Task 取消时的资源泄露：使用 `asyncio.TaskGroup` 管理

---

### Task 12：接入 OMS 下单回调（真实下单 + 成交回调）

**目标**：策略信号触发后走 OMS 真实下单，成交后回调 `runner.on_fill()`

**实施步骤**：

| 步骤 | 操作 | 文件 |
|------|------|------|
| 3.1 | 修改 `get_strategy_runner()` 注入 `oms_callback`、`event_callback`、`killswitch_callback` | `trader/services/strategy_runner.py` |
| 3.2 | 实现 `_oms_callback()`：Signal → 下单参数映射 | 同上 |
| 3.3 | 使用 `get_symbol_step_size()` 与 `quantize_by_step_size()` 做数量合法化 | `trader/core/application/oms.py` |
| 3.4 | 调用 `BinanceSpotDemoBroker.place_order()` 执行真实单 | `trader/adapters/broker/binance_spot_demo_broker.py` |
| 3.5 | 处理错误（余额不足、最小名义金额、精度错误）并发布 `strategy.order.rejected` 事件 | `trader/services/strategy_runner.py` |
| 3.6 | 订单与成交写入 storage（orders/executions） | `trader/storage/in_memory.py` |
| 3.7 | 实现 WebSocket 成交回调监听，更新订单状态并调用 `runner.on_fill()` | 新增 `trader/services/fill_callback_handler.py` |

**验收标准**：
- 策略触发信号后 `/v1/orders` 可看到订单
- 有成交时策略收到 `on_fill` 回调
- 拒单原因可追踪

**风险点**：
- 下单与成交回调的幂等性：依赖 `cl_ord_id` + `exec_id` 去重
- 乱序处理：使用 Event Log 作为真理来源

### Task 14：自动交易安全闸门

**目标**：默认安全关闭，显式开启后才能真实下单，Fail-Closed

**实施步骤**：

| 步骤 | 操作 | 文件 |
|------|------|------|
| 5.1 | 添加环境开关 `LIVE_TRADING_ENABLED=true` | `trader/config.py` 或 `.env` |
| 5.2 | 在 `oms_callback` 开头校验开关，关闭时拒绝下单 | `trader/services/strategy_runner.py` |
| 5.3 | 实现交易规则校验（stepSize、minQty、minNotional） | `trader/core/application/oms.py` |
| 5.4 | 获取交易所规则（从 Binance `/exchangeInfo`）并缓存 | `trader/adapters/broker/binance_spot_demo_broker.py` |
| 5.5 | KillSwitch L1 禁新仓时拒绝新订单 | `trader/services/killswitch.py` |
| 5.6 | KillSwitch L2/L3 停止策略 | 同上 |
| 5.7 | 拒单发布 `strategy.order.rejected` 事件，带明确 reason | `trader/services/strategy_runner.py` |

**验收标准**：
- 开关关闭时任何信号不发实单
- 开关开启且条件满足时可正常下单
- KillSwitch 触发后行为符合级别定义

**风险点**：
- 开关状态变更的竞态：使用锁保护
